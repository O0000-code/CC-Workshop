### P1

**[P1-1] §10.4「V1.5/V2 路线图占位」与 §9.3「不为未来铺垫」语义微冲突**

- **证据**：§9.3 末段"V1 的精神是'少做但做透'。任何'为未来铺垫'的代码 / 视觉 / 命名都需要先证明'V1 周期内会被使用'，否则不引入。" 但 §10.4 列出 6 项 V1.5/V2 演进方向（Agent / 自动更新 / Glama / 上游分类 / 滚动加载 / i18n），即使带"不作承诺"前缀，仍会让下游 spec / 实施 SubAgent 形成"路线图心智"——为 6 项中任一项暗中保留扩展点。
- **修订建议**：删 §10.4 整段，仅在 OQ-1~OQ-8 中保留 V1.5/V2 评估锚。理由：路线图本应是 PRD 之外的产品决策，不应在"V1 PRD"末尾压入 6 行"未来"；OQ 已足够把延后项跟踪起来。删除约 14 行。

**[P1-2] §10.1 P0 风险表与 §5 / §7 章节重复**

- **证据**：§10.1 是一张 21 行表，每条 R-N 都附"产品层处理"+"章节锚"，但产品层处理几乎全部已在 §5（关键交互）、§7（闭环定义）中详述过——表本身是反向索引。`plan-document-style.md` 反模式："同一信息在多个任务卡重复：错。抽到顶层'共同必读'或'风险登记'。"
- **修订建议**：保留 §10.1 表头与"完整 R-1~R-20 见 02_risk_distillation.md"一句指向，删除 21 行的处理描述（处理本身已在前面章节）；保留章节锚做反向索引即可。压缩约 18 行。

**[P1-3] §1.3「现有数据模型已为 marketplace 留好 hook」措辞可能误导**

- **证据**：§1.3 三个外部条件其三："Ensemble 现有数据模型已为 marketplace 留好 hook——`Skill` / `McpServer` 类型已有 `installSource` / `marketplace` / `pluginId` 字段（虽然当前只服务于 Claude Code plugin 来源），扩展第三态 `'marketplace'` 是顺势而为，不是推倒重来"。这描述的是事实（H §3 调研已确认），但措辞"留好 hook"对未读 H 的下游 SubAgent 可能被反向理解为"PRD 在为某种实现路径预设"——与 V1 PRD"不写技术实现"的角色定位有张力。
- **修订建议**：把"已为 marketplace 留好 hook"改为"已包含 plugin 来源标识字段（`installSource: 'local' | 'plugin'`），扩展第三态 `'marketplace'` 是同语义自然延伸"。措辞向"事实陈述"靠拢，不向"路径预设"靠拢。约 1 行改动。
