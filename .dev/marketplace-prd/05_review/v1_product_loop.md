### 专项详注（每项 1-3 行）

- **专项 1**：异步触发（§3.2 [5.5] "立即异步发起"）+ 单项作用域（§7.2 "仅该新装项不全量重跑"）+ 失败不阻断（§7.2 "资源仍出现在列表中"）三条都被说清，settings flag 链路（R-22）也被显式注明 V1 启用。唯一缺失是失败时的用户感知通道，进 P0-4。
- **专项 2**：四态机本身完整（§5.5 / V1 In #8）。但状态机是"画在屏幕上的"——它没说 Marketplace 列表如何**知道某项是 Installed**——这是引擎层缺口，进 P0-1。重装态（§5.6 三选项）覆盖完整。
- **专项 3**：这是 V1 闭环中**最大的产品断点**。Marketplace 与 Skills 是两个列表共享同一资源池，PRD 没定义两者的状态同步契约——会出现"我从 Skills 删了，Marketplace 仍说 Installed"或反过来的不一致。进 P0-1 的孪生问题。
- **专项 4**：上游不可达（缓存 / EmptyState）+ 网络失败（按钮变红）+ 上游错误（error banner）+ 同名碰撞（Confirm Modal）四类有解。缺三处：安装中网络中断（P1-1）、Auto-Classify 失败（P0-4）、列表与详情失败态同步（P0-5）。
- **专项 5**：PRD 说"卸载等同 Skills More menu Delete → Trash → 30 天清除"，但完全没定义 30 天窗口期内重装的语义。这是 macOS-native 用户最高频的"误删恢复"路径。进 P0-2。

## 发现清单（按 P0 / P1 / P2 排序）

> **分布概览**：P0 = 5 条（集中在状态层 SSoT 与跨模块同步契约）；P1 = 6 条（覆盖异常分支与产品语义边界）；P2 = 4 条（表述清晰度与扩展信号）。所有发现的"位置"字段都引用 PRD V1 的具体章节锚，便于 V2 修订定位。

### P0（必须在 V2 修订中处理）

**[P0-1] Marketplace 列表"已装态"判定的反向同步契约缺失**

- **位置**：§5.3、§7.4、§3.2 [5]、V1 In #8
- **问题**：PRD 反复说"Marketplace 列表内自有状态判定"（§7.4），但从未定义**判定基于什么**。是基于物理路径 `~/.ensemble/skills/<name>/` 的存在性？基于 `data.json.skillMetadata` 中的 entry？基于内存中的"installed set"？三者在用户从 Skills 列表 Delete 后会出现不一致：物理路径可能进 Trash 仍存在、metadata 可能保留也可能清除。
- **影响**：P0 闭环断裂——用户从 Skills 删除一个 marketplace 资源后回到 Marketplace，可能仍看到 `Installed` 状态（与 R-1 同名碰撞误判同根问题）。这是 PRD 已识别的 R-1（"`dest_skill_path.exists()` 短路把已存在误算作已导入"）的产品层重演。
- **修订建议方向**：在 §7.4 显式定义"已装"判定的产品契约——例如"Marketplace 列表的 `Installed` 状态当且仅当 `~/.ensemble/skills/<name>/` 存在 **且** `data.json.skillMetadata` 中 entry 存在 **且** 未在 Trash"。给 V2 一个明确的 single source of truth。

**[P0-2] Trash 30 天期间的重装语义完全未定义**

- **位置**：§7.6、§7.7
- **问题**：§7.6 说"卸载走 Skills 列表 More menu Delete → 进 Trash → 30 天后清除"。但 PRD 没说 30 天期间用户从 Marketplace 重新点 Install 会发生什么——(a) 是触发"同名碰撞" Modal（§5.6）？(b) 直接覆盖 Trash 中条目？(c) Trash 中条目被忽略、新装不感知 Trash？三种行为产品语义完全不同。
- **影响**：P0 闭环断裂——这是用户的真实路径（"我误删了一个 skill 想重装"），PRD 留白会让实施 SubAgent 任意决策，破坏 Trash 模型的一致性。
- **修订建议方向**：在 §7.6 / §7.7 加一节"卸载与重装的时间边界语义"——明确"Trash 期间从 Marketplace 重装"等价于"从 Trash 恢复"还是"覆盖式新装"，并给出对应 metadata 持久化策略（继承 / 丢失）。

**[P0-3] stdio MCP env vars 的填写、保存、生效链路断裂**

- **位置**：§3.2 [5]、§5.4、§7.5
- **问题**：PRD 说"stdio 类装完按钮变 `Installed (configure env)`"+"详情面板'配置区'显示需填字段"。但完全没说：(a) 用户**点 `Installed (configure env)` 后做什么**？（按钮 disabled 还是再次跳到详情？）(b) 配置区**是否可编辑**？是输入框还是只读预览？(c) env vars 填完后**保存动作**是什么？显式 Save 按钮还是即时持久化？(d) env 缺失或填错**反馈如何**？(e) env 填完后按钮状态**是否变化**（如从 `Installed (configure env)` → `Installed`）？
- **影响**：P0 闭环断裂——MCP 闭环 #6 的核心交付物（"装完能用上"）取决于这条链路；V1 In #3、V1 In #4 都依赖此能闭环。
- **修订建议方向**：§5.4 详情面板节奏的"元数据栏"段加一节"stdio 配置区交互契约"，定义编辑入口、保存动作、按钮状态映射、错误反馈四件事的产品行为（不写技术实现）。或在 §7.5 单独加一节"stdio MCP 装后→填→可用"路径。

**[P0-4] Auto-Classify 失败的用户感知反馈缺失**

- **位置**：§3.2 [5.5]、§7.2
- **问题**：§7.2 说"分类失败不阻断装成功——资源仍出现在列表中，只是无 Category / Tag；用户可手动改"。但用户**从哪里知道分类失败**？资源就这样静默以"未分类"出现？没有 Toast / inline 标记 / 详情面板提示？用户只能猜"是不是因为我没分配 Category"。
- **影响**：P0 闭环断裂——Auto-Classify 是 V1 In #5 的核心承诺，失败时静默会让用户误判产品本身分类不准。这是 R-15"装了但分类没跟上"的孪生场景。
- **修订建议方向**：§7.2 加一句产品契约——"分类失败时该项 row 显示一个非阻断的 inline 提示（如 'Auto-classify failed, click to assign manually'），用户点击展开手动分类入口"。具体形态留 spec，但产品反馈必须有。

**[P0-5] Marketplace 列表项 Install 按钮失败状态与详情按钮失败状态的同步未定义**

- **位置**：§5.5、§5.7
- **问题**：§5.5 说"失败时按钮就地变红 + 一行短文 `Installation failed. Retry?`"——但"按钮"指的是详情面板按钮还是列表项按钮？两处都接受点击（V1 In #8 / §5.5 末段），但失败后两处状态如何同步？如果用户在列表项触发安装、失败后切去其他页面、再回来，列表项按钮还显示红色失败态吗？PRD 留白。
- **影响**：P0 闭环断裂——失败态如果不持久化、用户感知会"消失但没成功"。
- **修订建议方向**：§5.5 / §5.7 加一句——"失败态绑定到列表项 entry 而非视图位置，跨页面切换持久化直到用户点 Retry 或下一次成功安装"。

### P1（应在 V2 修订中处理或 Open Questions 跟踪）

**[P1-1] "安装中"网络中断的产品行为未单独定义**

- **位置**：§5.5、§5.7
- **问题**：§5.5 列了"网络失败"= Installation failed + Retry。但"开始下载到一半网络断开"是否同等待"网络失败"处理？还是认为已落到 `~/.ensemble/` 部分文件需清理？产品反馈是同款 `Failed. Retry?` 还是 `Connection lost. Resume?` 不同语义？
- **修订建议**：§5.7 表格补一行"安装中网络中断"，至少表明"等同安装失败处理 + 自动清理半成品"或"V1 接受边界，与失败同款反馈"。

**[P1-2] §7.1 三段式与 §3.2 八步旅程对 Auto-Classify 的层级表达不一致**

- **位置**：§7.1 表格 vs §3.2 [5.5]
- **问题**：§7.1 把闭环描述为"装 → 选 → 部署"三段，Auto-Classify 不在表中；§3.2 把它列为子步 [5.5]"自动分类（异步、不阻塞）"。一个把它视作"装"的子动作，一个把它视为独立步。下游 spec / tech plan 读者会困惑"自动分类是装的一部分还是装后第二阶段"。
- **修订建议**：在 §7.1 表格"装"列的"用户感知"格补一句"分类异步出现，不阻断后续 Scene 编排"。或显式说"Auto-Classify 隶属装阶段的尾巴，不是独立一段"。

**[P1-3] 卸载 marketplace 资源时是否清理上游来源记录未明**

- **位置**：§7.6、§7.7
- **问题**：§7.7 说"metadata 持久化在 `data.json.skillMetadata`"+"卸载再装丢 metadata"。但 marketplace 来源（owner/repo/skill-name）是否在卸载时一同被清除？如果保留，会成为孤儿记录；如果清除，再装时无法继承"上次从哪里装的"信息。
- **修订建议**：§7.7 / §10.3 OQ-5 显式注明 V1 行为——"卸载等同清除全部 metadata 含 upstream 来源；再装走全新路径"，或"upstream 来源保留以备恢复"。

**[P1-4] OAuth 类 HTTP MCP "在 Claude Code 内 /mcp 完成认证"动作的反馈回流缺失**

- **位置**：§3.2 [4]、§5.4、§7.5

- **修订建议**：§7.5 表格"HTTP / OAuth"行补一句"V1 不感知 Claude Code 内 OAuth 状态，提示文案恒显；V1.5 评估认证状态回流"，把边界点透。

**[P1-5] 同名碰撞 Modal 的"Replace"行为对 metadata 的影响未明**

- **位置**：§5.6
- **问题**：§5.6 说"`Replace` 用 marketplace 来源覆盖本地（旧版本不保留备份）"。但"覆盖"是否仅覆盖文件内容、保留 Category / Tag / 自定义图标？还是连 metadata 一起覆盖？"备份"特指物理文件还是含 metadata？
- **修订建议**：§5.6 加半句——"Replace 等价于物理拷贝覆盖 + metadata 继承（保留 Category / Tag / 自定义图标）"。

**[P1-6] Refresh 按钮强制刷新缓存与"用户在用列表"的并发行为未明**

- **修订建议**：§5.7 增加一行"Refresh 触发时"的视觉反馈描述。

## 全局观察（闭环视角的横向 pattern）

下面三条是横向归纳，本身不进发现清单，但解释了为什么 5 条 P0 集中在状态层而非决策层：

- **状态判定的 single source of truth 缺失**：P0-1 / P0-2 / P0-5 都源于同一根因——PRD 描述了"用户看到什么"但没定义"系统如何决定该看到什么"。Marketplace 列表的"已装态"、Trash 中条目的"还存在态"、失败按钮的"持久化态"都缺少 SSoT 声明。这是产品契约层的缺口，不是 spec 层的实现细节。
- **跨模块状态流的契约写得很弱**：Marketplace ↔ Skills、Skills ↔ Trash、详情面板 ↔ 列表项三对模块边界都需要"反向同步契约"。PRD 在每个模块各说各话，没有把"当 X 在 A 模块发生时，B 模块如何感知与反应"显式连成图。R-1 / R-3 / R-32 都是该缺口的局部投影。
- **Auto-Classify 与 stdio env 配置是闭环的两个"长尾子流程"**：它们都是"装完后异步发生"的事——Auto-Classify 是被动触发、env 是用户主动填。两者都在 PRD 中被简化为"会发生" + "用户可手动改"——但**用户从哪里看到失败 / 从哪里进入 / 改完后如何感知生效**这三件事都未连完。这是 V1 闭环承诺"装完即用"的核心兑现路径。

## 总体判断

**Pass with revisions**

PRD V1 的**高位骨架成立**：

- 三段式契约（装 → 选 → 部署）边界清晰（§7.1）
- `installSource` 三态扩展决策（D-9 / §7.3）解决了 R-2 / R-4 / R-11 三条 P0
- Auto-Classify 单项异步触发（D-8 / §7.2）解决了 R-15
- Marketplace 资源不沉底排序、与本地资源平等参与 Scene 编排（§7.3 / V1 In #7）守住了用户原话"装到管理路径，能直接分发使用"
